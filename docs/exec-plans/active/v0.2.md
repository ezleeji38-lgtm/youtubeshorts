# V0.2 실행 계획

> 상태: 진행 중 · 목표: V0.1 곡 단위 클립 생성기 → 플레이리스트 워크플로우로 확장

## V0.2 목표

V0.1은 **"곡 한 개의 한 구간 → 클립 한 개"**까지 가능했다. V0.2는 다음 두 가지를 목표로 한다.

1. **외부 렌더 API 의존 제거** → 로컬 FFmpeg + ASS 자막 합성으로 자립
2. **플레이리스트 워크플로우** → Playlist(15곡) → Song → Short 계층 구조

## 결정 로그

### 2026-04-30 · libass 미설치 발견 (V0.2 #1)

- **상황**: macOS Homebrew 기본 `ffmpeg` 8.1이 libass·libfreetype 빠진 빌드로 출고됨. 자막 burn-in 불가.
- **결정**: ① 사용자에게 `brew install ffmpeg-full` 안내 ② 런타임 사전 검증 추가 (`render._check_ffmpeg_capabilities()`).
- **대안**: drawtext 필터로 폴백 (보류 — libass 자막 품질이 압도적, ffmpeg-full 설치 한 번이면 끝나는 문제).
- **남은 위험**: 동생이 Windows면 별도 설치 가이드 필요.

### 2026-05-02 · `Short.song_id`를 nullable로 (V0.2 #2)

- **상황**: V0.1에서 만들어진 ad-hoc Short 레코드가 있을 수 있음. 새 Song FK가 강제되면 호환 깨짐.
- **결정**: `song_id`를 nullable. `/api/shorts/`로 만든 단독 클립은 song_id=NULL로 그대로 둠.
- **트레이드오프**: 데이터 정합성이 약해지지만, 단일 사용자 로컬 도구라 영향 미미.

### 2026-05-02 · `is_shorts_pick` 자동 해제 (V0.2 #2)

- **상황**: 한 플레이리스트(15곡) 안에서 `is_shorts_pick=True`인 곡은 단 하나여야 함.
- **결정**: `PATCH /api/songs/{id}`에서 `is_shorts_pick=True`로 켤 때, 같은 `playlist_id`의 다른 곡은 자동으로 False로 갱신 (한 트랜잭션 안에서 atomic).
- **대안**: DB 레벨 partial unique index — SQLite에서도 `WHERE is_shorts_pick=1`로 가능하지만 SQLAlchemy declarative에서 정의 번잡. 애플리케이션 레벨 보장으로 충분.

### 2026-05-02 · 마이그레이션은 V0.3까지 보류

- **상황**: V0.2 #2에서 `Short.song_id` 컬럼이 추가됨. SQLAlchemy `create_all()`은 신규 테이블만 만들고 기존 테이블에 컬럼을 추가하지 않음.
- **결정**: 단일 사용자 로컬 도구라 영향 적고, V0.1을 실제로 돌려본 사람이 아직 없음(libass 이슈로). 일단 `rm storage/coolvibeply.db`로 재생성하라고 README에 안내. Alembic 도입은 V0.3.
